Aufgabe 09b

Component State mit Callback-Props

🎯 Lernziel

Du löst dieselbe Aufgabe wie in 09, aber mit vollständig kontrollierten Komponenten. Der Parent hält den gesamten State – die Children haben kein internes useState mehr. Jeder Wert und jedes Event wird über Props gesteuert.

🔧 Konzepte

value + onChange Controlled Components Lifting State Up interface Props ChangeEvent Parent→Child ohne internen State

📋 Aufgabe

  1. 1. Erstelle eine Seite Calculator.tsx – sie ist der "Controller".
  2. 2. Baue die Komponenten ein – aber alle Children sind vollständig kontrolliert (kein useState im Child, nur value + onChange Props).
  3. 3. Jede Child-Komponente bekommt ihren aktuellen Wert per value-Prop und meldet Änderungen per onChange-Callback.
  4. 4. Der Parent hat alle Handler-Methoden, die den State aktualisieren und die Berechnung triggern.
  5. 5. Verwende useEffect (optional), um auf Änderungen zu reagieren und das Ergebnis automatisch zu berechnen.

📌 Der Unterschied zu 09 — Trade-off

Aspekt 09 (Child mit useState) 09b (Controlled)
Child-State Eigenes useState + Callback-Prop Kein State, nur value + onChange
Parent-Code State + Callback-Props + Handler State + onChange + Handler (direkter)
Datenfluss Child useState → Callback → Parent State → Prop Parent State → Prop → onChange → Parent State
Wiederverwendbarkeit Kann standalone ohne Parent Braucht immer einen Parent mit State

⭐ Empfehlung: Variante 09b (Controlled) für diese Übung. Klarerer uni-direktionaler Datenfluss, Single Source of Truth im Parent, weniger Überraschungen.

💡 Hinweise – Controlled Components in der Praxis

So funktioniert das Controlled-Component-Pattern:

// Child-Komponente hat KEINEN eigenen State:
interface CategorySelectProps {
  value: string;
  onChange: (cat: string) => void;
}

function CategorySelect({ value, onChange }: CategorySelectProps) {
  return (
    <select value={value}
            onChange={e => onChange(e.target.value)}>
      <option value="" disabled>Bitte wählen...</option>
      {categories.map(c => (
        <option key={c} value={c}>{c}</option>
      ))}
    </select>
  );
}

// Parent gibt value und onChange:
<CategorySelect value={selectedCategory}
                onChange={handleCategoryChanged} />

💡 Der gesamte State lebt im Parent. Das Child ist ein reines "Render"-Element ohne eigene Logik.

🔧 Parent-Seite (Code-Gerüst)

Vergleiche: 09 (Child-State) vs. 09b (Controlled):

// 09 — Child hat useState + Callback
<CategorySelect onCategoryChanged={handleCategory} />
<QuantityInput onQuantityChanged={handleQuantity} />
<DualInput onValueChanged={handleFee} />

// 09b — Child hat keinen State, Parent steuert alles
<CategorySelect value={selectedCategory}
                onChange={setSelectedCategory} />
<QuantityInput value={quantity}
               onChange={setQuantity} />
<DualInput value={fee}
           onChange={setFee} />

const [selectedCategory, setSelectedCategory] = useState("");
const [quantity, setQuantity] = useState(0);
const [fee, setFee] = useState(2.5);
const [result, setResult] = useState<number | null>(null);

useEffect(() => {
  if (selectedCategory && quantity > 0) {
    setResult(quantity * (1 + fee / 100));
  } else {
    setResult(null);
  }
}, [selectedCategory, quantity, fee]);

📌 Vorteil: Keine Handler-Funktionen nötig – setSelectedCategory kann direkt als onChange übergeben werden, weil die Signatur (value: string) => void mit useState kompatibel ist.

🏁 Erforderliche Child-Schnittstellen

Jede Child-Komponente braucht value + onChange Props (statt eigenem useState + Callback):

// CategorySelect (controlled)
interface CategorySelectProps {
  value: string;
  onChange: (cat: string) => void;
}

// QuantityInput (controlled)
interface QuantityInputProps {
  value: number;
  onChange: (val: number) => void;
  disabled?: boolean;
}

// DualInput (controlled)
interface DualInputProps {
  value: number;
  onChange: (val: number) => void;
}

// Parent-Seite Calculator.tsx
<CategorySelect value={selectedCategory}
                onChange={handleCategoryChanged} />
<QuantityInput value={quantity}
               onChange={setQuantity}
               disabled={!selectedItem} />
<DualInput value={fee} onChange={setFee} />

const [selectedCategory, setSelectedCategory] = useState("");
const [selectedItem, setSelectedItem] = useState("");
const [quantity, setQuantity] = useState(0);
const [fee, setFee] = useState(2.5);

const handleCategoryChanged = (cat: string) => {
  setSelectedCategory(cat);
  setSelectedItem(""); // Reset Item
};

💡 Die alten Callback-Props (onCategoryChanged, onQuantityChanged) entfallen – die Komponente unterstützt nur noch das Controlled-Pattern.

💡 Tipps

  • Das Controlled-Pattern ist die idiomatischste React-Art, Formulare zu bauen.
  • Der Parent ist die "Single Source of Truth" – keine versteckten State-Updates im Child.
  • useEffect kann genutzt werden, um auf State-Änderungen zu reagieren, ist hier aber optional (Berechnung kann auch im Handler passieren).
  • Setter von useState können direkt als onChange übergeben werden, wenn die Typen kompatibel sind.

✅ Akzeptanzkriterien

  • Children haben kein useState – nur value + onChange Props
  • Parent hält den gesamten State
  • Berechnung läuft über Handler oder useEffect
  • Bei Kategorie-Wechsel wird das Item zurückgesetzt
